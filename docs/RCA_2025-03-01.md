## 3. What Was Correct in the Original RCA
- Power/interlock sequencing and command pacing were key constraints.
- Bus congestion and stale RX traffic were major failure modes.
- Vendor sequencing (`initializeMotorsWithoutMotion`, `initializeMotors`, `enableXYZ`) was the correct reference model.

---

## 4. Current Root Cause Focus
The dominant issue is no longer "unknown protocol" for core control.  
The dominant issue is reliability under sustained command traffic:

1. USB/CAN receive-path contamination from asynchronous traffic during multi-command operations.
2. Command burst pacing gaps that can produce transient no-response windows.
3. Incomplete parity between motor recovery and thermal/chiller recovery mechanisms.

---

## 5. Remaining Risks
1. Intermittent no-response streaks can still occur during aggressive operator-driven testing.
2. Motor path needs the same pacing/no-response auto-recovery structure already used for thermal/chiller paths.
3. Door homing still benefits from tighter parity with vendor-specific `doorSearchHome` behavior.

---

## 6. Current Conclusion
Control is no longer blocked by missing protocol discovery for core subsystems.  
The project is in a near-full-control state, with remaining work focused on comm robustness and recovery behavior rather than missing command maps.
